# Claude Code Agent Teams 심층 조사

> 공식 문서 기반. 2026-03-18 기준.

---

## 1. Agent Teams vs 서브에이전트

| 구분 | 서브에이전트 | Agent Teams |
|------|-------------|-------------|
| 컨텍스트 | 독립 윈도우, 결과만 반환 | 독립 윈도우, 완전 독립 |
| 통신 | 부모에게만 보고 | 팀원끼리 직접 메시지 |
| 조율 | 메인 에이전트가 관리 | 공유 태스크 리스트로 자기 조율 |
| 비용 | 낮음 (결과 요약) | **~7배** (각 팀원이 별도 인스턴스) |
| 적합 | 결과만 필요한 포커스 작업 | 토론/협업이 필요한 복합 작업 |

**핵심 차이**: 서브에이전트는 "위임", Agent Teams는 "협업"

## 2. 활성화 및 시작

### 활성화 (기본 비활성화)
```json
// settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

또는 셸에서:
```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

**요구사항**: Claude Code v2.1.32+

### 팀 시작
자연어로 요청:
```
CLI 도구를 설계할 건데, 에이전트 팀을 만들어줘:
- UX 담당 1명
- 기술 아키텍처 담당 1명
- 반대 의견 제시 담당 1명
```

Claude가 팀 생성 → 팀원 스폰 → 작업 분배 → 결과 종합

## 3. 팀 제어

### 디스플레이 모드

| 모드 | 설명 | 설정 |
|------|------|------|
| **in-process** (기본) | 한 터미널에서 Shift+Down으로 팀원 순환 | `"teammateMode": "in-process"` |
| **tmux** | 각 팀원 별도 pane | `"teammateMode": "tmux"` |

### 팀원 직접 대화
- **Shift+Down**: 팀원 순환
- **Enter**: 팀원 세션 보기
- **Escape**: 현재 턴 중단
- **Ctrl+T**: 태스크 리스트 토글

### 팀원/모델 지정
```
3명의 팀원으로 팀 만들어줘. 각각 Sonnet 사용.
- 보안 리뷰어
- 성능 감사자
- 테스트 커버리지 전문가
```

### Plan 승인 요구
```
auth 모듈 리팩토링할 architect 팀원을 만들어줘.
변경 전에 plan 승인을 받도록 해줘.
```

### 태스크 관리
- 태스크 상태: Pending → In Progress → Completed
- 태스크 의존성 지원 (의존성 완료 전 클레임 불가)
- 파일 락킹으로 경합 방지

### 팀원 종료 & 정리
```
researcher 팀원 종료해줘
```
```
팀 정리해줘   # 반드시 lead에서 실행
```

### Quality Gate 훅
```json
{
  "hooks": {
    "TeammateIdle": [{
      "hooks": [{
        "type": "command",
        "command": "./scripts/quality-check.sh"
      }]
    }],
    "TaskCompleted": [{
      "hooks": [{
        "type": "command",
        "command": "./scripts/verify-task.sh"
      }]
    }]
  }
}
```

## 4. 아키텍처

### 구성 요소
| 컴포넌트 | 역할 |
|----------|------|
| **Team Lead** | 팀 생성, 팀원 스폰, 작업 조율 |
| **Teammates** | 각각 독립 Claude Code 인스턴스 |
| **Task List** | 공유 작업 목록 (클레임/완료) |
| **Mailbox** | 에이전트 간 메시지 시스템 |

### 통신
- 메시지 자동 전달 (lead 폴링 불필요)
- `message`: 특정 팀원에게
- `broadcast`: 모든 팀원에게 (비용 주의)
- Idle 시 자동으로 lead에 알림

### 팀원이 받는 것
- 독립 컨텍스트 윈도우
- CLAUDE.md, MCP, Skills (프로젝트 컨텍스트)
- Lead의 스폰 프롬프트

### 팀원이 받지 못하는 것
- Lead의 대화 히스토리
- Lead의 이전 추론

### 권한
- Lead의 권한 설정 상속
- 스폰 후 개별 변경 가능
- 스폰 시 개별 설정 불가

## 5. 토큰 비용 (~7배)

### 비용 구조
3명 팀원 일반 작업:
- 단일 세션: ~100k 토큰
- 3명 팀: ~700k 토큰 (**7배**)

### 비용 원인
1. 각 팀원 독립 컨텍스트 윈도우
2. 각 팀원 별도 API 호출
3. 팀 조율 오버헤드
4. 결과 요약 없음 (서브에이전트와 다름)

### 비용 절감
1. **Sonnet 사용** (Opus 대신)
2. **3-5명 적정 규모** 유지
3. **스폰 프롬프트 간결하게** (CLAUDE.md는 자동 로드됨)
4. **작업 완료 후 즉시 정리**
5. **태스크 크기 적정화** (팀원당 5-6개)

## 6. 실전 사용 사례

### 사례 1: 병렬 코드 리뷰
```
PR #142를 리뷰할 팀 만들어줘:
- 보안 관점 리뷰어
- 성능 영향 확인자
- 테스트 커버리지 검증자
```
각 관점에서 독립 리뷰 → Lead가 종합

### 사례 2: 경쟁 가설 조사
```
앱이 메시지 한 번 후 종료됨.
5명의 에이전트 팀을 만들어서 다른 가설을 조사하게 해줘.
서로의 이론을 반증하려고 토론하게 해줘.
```
앵커링 편향 방지 → 실제 근본 원인 발견

### 사례 3: 기능 병렬 개발
```
실시간 알림 시스템 구축. 4명 병렬:
- 백엔드 이벤트 핸들러
- WebSocket 연결 관리
- 프론트엔드 알림 UI
- DB 스키마 및 마이그레이션
```
각 팀원이 다른 파일 소유 → 충돌 없는 병렬 작업

## 7. 베스트 프랙티스

1. **충분한 컨텍스트 제공**: 스폰 프롬프트에 작업별 세부사항 포함
2. **3-5명 적정 규모**: 그 이상은 조율 오버헤드 > 이득
3. **태스크 크기**: 30분~2시간, 자체 완결, 테스트 가능
4. **파일 충돌 방지**: 팀원별 다른 파일/디렉토리 소유
5. **리서치/리뷰부터 시작**: 코딩 전에 비파괴적 작업으로 연습
6. **주기적 모니터링**: 30분마다 진행 상황 확인, 방향 교정
7. **Lead가 정리**: 팀원이 정리하면 리소스 불일치 위험

---

## 8. 제한사항

1. **세션 재개 시 in-process 팀원 복원 불가** → 새 팀원 스폰 필요
2. **태스크 상태 지연**: 팀원이 완료 표시 안 할 수 있음
3. **종료 느림**: 팀원이 현재 요청 완료 후 종료
4. **세션당 1팀만**: 현재 팀 정리 후 새 팀 생성
5. **중첩 팀 불가**: 팀원이 자체 팀 생성 불가
6. **Lead 고정**: 리더십 이전 불가
7. **스폰 시 권한 일괄**: 개별 권한 설정 불가
8. **Split pane 제한**: VS Code 터미널, Windows Terminal, Ghostty 미지원
