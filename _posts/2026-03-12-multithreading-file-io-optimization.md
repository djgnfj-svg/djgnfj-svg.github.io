# 배경

반도체 IP 회사에서 QA를 하고 있었다. 팀에서 쓰는 검증 스크립트가 있었는데, 텍스트 파일들을 하나씩 읽어서 오류를 검사하고 excel로 만드는 코드였다.

원래 SW팀에서 하던 일이었는데, 이게 HW팀으로 넘어오면서 검증해야 할 파일이 약 120배 정도 늘어났다. 십수 개 처리하던 스크립트가 수백 개를 처리해야 하는 상황이 되었다.
한 번 돌리는 데 9시간 퇴근 전에 돌려놓고 다음 날 출근해서 결과를 확인하는 수준이었다.

하지만 그전에는 1시간 단위로 나오는 일이 갑자기 9시간 걸리는 것이 마음에 들지 않았고 이게 만약에 에러가 있으면 또 prom 굽고 다시 측정을 돌려야하는데 이렇게 되면 다음 검증이 2일 뒤로 미루어진다. 그래서 팀장님께 말씀드리고 코드를 리팩토링 하기로 하였다.

# 문제 분석

해결방법은 간단했지만 과정은 쉽지 않았다. 코드가 땜질 덩어리여서 1800줄 이상이었고 리팩토링까지 겸해서 처리해야 했다. 리팩토링은 차지하고 결국에 중요한 부분은

```python
# Before: 순차 처리
for file_path in file_list:
    content = read_file(file_path)  # 파일 I/O (느림)
    check_errors(content)           # 검사 (빠름)
```

이러한 부분을

```python
# After: 멀티스레딩 병렬 처리
from concurrent.futures import ThreadPoolExecutor
import os

max_workers = os.cpu_count() + 2

with ThreadPoolExecutor(max_workers=max_workers) as executor:
    results = executor.map(process_file, file_list)
```

이렇게 바꾼 것이다.

파일 읽는 데 대부분의 시간이 걸리고, 실제 검사 로직은 금방 끝났다. 전형적인 **I/O bound** 작업이었다.

핵심은 파일을 읽는 동안 CPU는 그냥 기다리고 있는 부분에서 병목이 발생한 것이였다.


# 왜 멀티스레딩인가?

Python에는 GIL(Global Interpreter Lock)이라는 게 있어서 한 번에 하나의 스레드만 Python 코드를 실행할 수 있다. (정확히는 GIL 자체는 잠금 장치를 칭하는 명칭)

그래서 CPU를 많이 쓰는 작업(CPU bound)에서는 멀티스레딩을 써도 의미가 없다. 스레드가 10개여도 결국 한 번에 하나만 실행되니까.

근데 I/O bound는 다르다. 파일을 읽거나 네트워크 요청을 보내는 동안에는 **GIL이 해제**된다. 운영체제가 I/O를 처리하는 동안 다른 스레드가 실행될 수 있다는 뜻이다.

정리하면:
- **CPU bound** → 멀티스레딩 효과 없음 (멀티프로세싱 써야 함)
- **I/O bound** → 멀티스레딩 효과 있음 (GIL이 풀리니까)

이 스크립트는 파일 I/O가 병목이었으므로, 멀티스레딩이 딱 맞는 상황이었다.

**CPU를 많이 쓴다면 멀티 프로세스를 사용한다**

# 해결

CPU 코어 수에 2개를 더 추가해서 워커 수를 설정했다. I/O 대기 시간에 다른 스레드가 바로 치고 들어올 수 있도록 코어 수보다 살짝 여유를 둔 것이다.

지금 돌아보면 Python 3.8+ ThreadPoolExecutor의 기본값이 `os.cpu_count() + 4`이고, I/O bound 작업은 어차피 스레드가 대부분 대기 상태라 더 늘려도 됐을 것 같다. 당시에는 이 정도로도 충분했지만, 다음에는 파일 수에 맞춰서 `min(32, len(file_list))` 같은 설정도 고려해볼 것 같다.


# 결과

**9시간 → 30분. 약 95% 단축.**

CPU가 놀지 않고 계속 일하게 만든 것만으로 이 정도 차이가 났다. 파일 하나를 읽는 동안 다른 파일을 동시에 읽기 시작하니까, 수백 개의 파일을 처리하는 전체 시간이 극적으로 줄었다.


# 배운 것

- 느린 코드를 만나면 일단 **병목이 어디인지** 파악하는 게 먼저다. CPU인지 I/O인지에 따라 해결책이 완전히 달라진다.
- 글을 정리하면서 알아낸 건 병목을 파악하는 걸 **프로파일링** 이라고 한다고 했다. 심지어 툴도 있다. 처음에는 단순히 파일이 많아져서 파일 IO라는 생각에서 멀티쓰레딩을 썼는데 다음에는 프로파일링도 돌려 볼 수 있으면 좋겠다.
