# 하네스 엔지니어링

개념 설명은 안 할 예정이다.


# Level 1
## Claude.md 설정하기

하네스에 격언처럼 찾아볼 때마다 나오는데 바로 적용하는 게 아니라 LLM을 사용하면서 서서히 확장시키는 구조로 사용된다. 그럼 기존의 자동 작성하는 것과 뭐가 다른가?

결론부터 말하면 init으로 시작해도 되고 진행하면서 추가하는 게 핵심이다.

아래 5가지를 유의하면서 작성하려고 한다.

### 1. 분량 제한을 지켜라
너무 길면 안 된다. 만약에 지시가 있으면 뚜렷하게 적어라.

### 2. 구체적이고 실행 가능하게
이렇게 작성해야 한다.
- 모든 public 함수에 JSDoc 주석 필수

이렇게 작성하면 그냥 제맘대로 한다.
- 코드 품질을 높게 해줘

### 3. 스타일 규칙은 넣지 마라
린터 쓰자. 노이즈이기도 하고 비용 생각해 봤을 때 그냥 자동 포맷팅 해주는 걸 쓰는 게 좋을 듯하다.

또는 훅에 맡겨도 된다.

정리하면 다음처럼 하면 됩니다.
| CLAUDE.md에 넣을 것 | 린터/훅에 맡길 것 |
|---|---|
| 아키텍처 의존성 방향 | 들여쓰기, 세미콜론 |
| 금지 패턴 (예: `any` 타입 금지) | import 정렬 |
| 빌드/테스트 명령어 | 코드 포맷팅 |
| 비즈니스 로직 주의사항 | 네이밍 규칙 (린터로 강제 가능 시) |

### 4. 파일이 커지면 분리하자
예전에 무지성으로 추가하다가 2000줄이 넘는 경우가 있었습니다. 경고는 주지만 그냥 무시했었죠.

하지만 프로젝트가 엄청 커진다면? 어쩔 수 없이 커질 수 있습니다. 그럴 때는 `.claude/rules/` 디렉토리를 활용해서 주제별로 규칙을 나눠 관리할 수 있습니다. 각 파일은 `testing.md`, `api-design.md` 같이 서술적인 이름을 붙이고, 경로 기반 규칙(YAML 프론트매터의 paths 필드)으로 특정 파일에만 적용할 수도 있습니다.
```
my-project/
├── CLAUDE.md              # 핵심만 (200줄 이내)
├── .claude/
│   └── rules/
│       ├── testing.md      # 테스트 컨벤션
│       ├── api-design.md   # API 설계 규칙
│       └── security.md     # 보안 요구사항
```

### 5. 살아있는 문서로 만들어라
항상 claude.md 업데이트해줘 메모리에 저장해줘 이러지 말고 hook으로 자동으로 업데이트되도록 해야 합니다.

이런 사소한 부분을 맡기고 다시 봤을 때 자동으로 문서가 업데이트 되고 있는 게 하네스 엔지니어링의 정수라는 것 같습니다.
