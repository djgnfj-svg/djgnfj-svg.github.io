# 배경

로봇 관제 시스템을 웹으로 전환하는 프로젝트를 하고 있었다. 기존에는 현장에서 PyQt로 한 명이 직접 조종하던 구조였는데, 웹으로 바뀌면서 여러 사용자가 동시에 접속할 수 있게 되었다.

문제는 여러 사용자가 같은 로봇에 Move/Stop 명령을 동시에 보내면 위험하다는 것이었다. 실제 충돌 사고가 난 건 아니었지만, 설계 단계에서 이건 반드시 막아야 한다고 판단했다. 한 번에 한 명만 로봇을 제어할 수 있어야 했다.


# 설계: 세션 기반 잠금

## Session ID

브라우저에서 `crypto.randomUUID()`로 UUID v4를 생성하고 `localStorage`에 저장했다. 새로고침해도 유지되고, 탭을 닫으면 사라지지 않는다.

## 3가지 상태

- **available** — 아무도 제어 안 함 (초록 LED)
- **locked_me** — 내가 제어 중 (파란 LED + Ctrl 배지)
- **locked_other** — 다른 사람이 제어 중 (주황 LED + Other 배지)

## 제어권 획득 흐름

1. 사용자가 Ctrl 버튼 클릭 → `POST /api/control/acquire/{robotId}?session_id=xxx`
2. 서버가 `ROBOT_LOCKS` dict 확인 → 비어있으면 성공, 다른 사람이면 거부
3. 제어권 없이 Move/Stop 시도 → "No control — command rejected" 토스트


# 하트비트

클라이언트에서 서버로 15초마다 HTTP POST를 보낸다.

```
POST /api/control/heartbeat/{robotId}?session_id=xxx
```

서버는 이걸 받을 때마다 `last_activity` 타임스탬프를 갱신한다. 브라우저 탭이 열려있는 한 자동으로 계속 전송된다.


# 타임아웃: 이중 안전장치

여기서 고민이 있었다. 하트비트만으로는 부족한 시나리오가 있었다.

- **브라우저 강제 종료** → 하트비트가 끊긴다 → 서버가 60초 후 자동 해제
- **브라우저 열어놓고 방치** → 하트비트는 계속 간다 → 그런데 실제 조작은 없다

두 번째 케이스를 잡기 위해 클라이언트 측 타임아웃을 따로 뒀다.

| 계층 | 타임아웃 | 트리거 |
|------|---------|--------|
| 서버 | 60초 | 하트비트가 끊기면 발동 |
| 클라이언트 | 180초 (3분) | 실제 조작(Move/Stop 등) 없으면 자동 반납 |

Move/Stop을 할 때마다 3분 타이머가 리셋된다. 즉 실제로 로봇을 조종하고 있으면 절대 끊기지 않고, 방치하면 3분 후 알아서 반납된다.


# 기술 선택

| 요소 | 선택 |
|------|------|
| 잠금 저장소 | Python In-Memory Dict (`ROBOT_LOCKS = {}`) |
| 동시성 보호 | `threading.Lock()` |
| 통신 방식 | 일반 HTTP REST |
| 상태 동기화 | 3초 폴링 (`GET /api/control/status`) |

Redis 없음, WebSocket 없음. 단일 서버 + REST 폴링으로 심플하게 갔다.


# 왜 이 구조를 선택했는가

다른 방안도 고려는 했다.

- **Redis 분산 락** — 수평 확장과 서버 재시작에 안전하지만, 로봇 수가 수십 대 수준이라 오버엔지니어링이었다
- **WebSocket 브로드캐스트** — 실시간 동기화에 좋지만, MQTT가 이미 로봇 데이터용으로 있어서 프로토콜이 혼재되는 게 싫었다
- **MQTT 토픽으로 잠금** — 기존 인프라를 활용할 수 있지만, 잠금은 서버가 중앙에서 관리하는 게 안전하다고 판단했다
- **DB 기반 락** — 영속성이 좋지만, 애초에 DB 없이 JSON 파일 기반으로 동작하는 아키텍처였다

현재 구조가 적합한 이유는 명확했다. 로봇 수십 대 규모, 단일 서버, 복잡도 최소화. 그리고 Session ID 기반이라 나중에 Redis로 전환하기도 쉽다.


# 한계점과 개선 방향

만들고 나서 보이는 한계점들이 있다.

**서버 재시작 시 잠금 전부 소실.** In-Memory Dict라서 서버가 죽으면 모든 잠금이 날아간다. 재시작 직후 여러 명이 동시에 제어권을 획득할 수 있는 순간이 생긴다. 현재 JSON 파일 기반 아키텍처니까 잠금 상태도 파일에 쓰는 것도 방법이고, 규모가 커지면 Redis로 전환해야 한다.

**클라이언트 타임아웃은 신뢰 기반이다.** 180초 비활성 반납은 클라이언트가 자발적으로 하는 것이다. 클라이언트에 버그가 있으면 하트비트만 계속 보내면서 잠금을 영원히 잡을 수 있다. 서버 측에서도 "마지막 명령 시각"을 추적해서 직접 비활성 판단을 하는 게 더 안전하다.

**멀티 워커 시 무력화.** uvicorn을 `--workers 2` 이상으로 띄우면 프로세스가 분리되면서 `ROBOT_LOCKS` dict가 각 워커마다 따로 존재한다. 현재는 단일 워커라 문제없지만, 스케일업하면 Redis 전환이 필수다.

**60초 타임아웃이 길다.** 브라우저가 크래시된 사이에 로봇이 움직이고 있으면, 60초 동안 아무도 Stop을 보낼 수 없다. 물리 장비 안전 관점에서 30초 이하로 줄이고 하트비트 간격도 5초로 단축하는 게 낫다.

**강제 회수 수단 없음.** 비정상 상황에서 관리자가 잠금을 강제로 뺏을 방법이 없다. 관리자용 `POST /api/control/force-release/{robotId}` 엔드포인트가 필요하다.


# 배운 것

- 동시성 문제는 터진 다음에 잡으면 늦다. 특히 물리 장비를 제어하는 시스템에서는 설계 단계에서 선제적으로 막아야 한다.
- 하트비트 하나만으로는 "방치" 시나리오를 잡을 수 없다. 서버 타임아웃과 클라이언트 타임아웃을 이중으로 두니까 빈틈이 없어졌다.
- 오버엔지니어링을 피하는 것도 설계다. Redis, WebSocket 다 좋지만 현재 규모와 상황에 맞는 가장 단순한 방법을 고르는 게 중요하다.
- 동시에, 현재 설계의 한계를 인지하고 있어야 다음 스텝을 빠르게 밟을 수 있다.
